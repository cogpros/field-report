Database choice for Atlas

Decision doc — pick the primary datastore before sprint 0 starts. Six questions, then a final call. Answer in the textareas; export at the bottom.

Atlas is a multi-tenant analytics app. ~500 customers expected in year 1, ~10M events/day at steady state. We're choosing the primary datastore now and don't want to revisit for 18 months. Three candidates were shortlisted: Postgres, ClickHouse, and a Postgres + ClickHouse hybrid.

1Workload shape

Atlas is ~80% analytical reads (dashboards, cohort queries) and ~20% transactional writes (user config, account state). Reads are heavy aggregations across millions of rows. Writes are small and frequent.

Does this match your expectation?

If the read/write ratio is wrong, every downstream choice shifts.

2Option A — Postgres only

Pros. One database to operate. Mature tooling. Team knows it. Analytical queries possible via materialized views + partitioning.
Cons. At 10M events/day, aggregation queries will need careful indexing. Some dashboards will get slow. Vertical scaling has a ceiling.

Is one-database simplicity worth the query-tuning cost?

3Option B — ClickHouse only

Pros. Built for analytical reads at this scale. Aggregations are milliseconds. Compression is excellent.
Cons. Transactional writes are awkward. No real foreign keys. Team has zero operational experience with it. Migrations are nontrivial.

Is the analytical performance worth the operational unknowns?

4Option C — Postgres + ClickHouse hybrid

Pros. Postgres for config + state. ClickHouse for event analytics. Each tool used for what it's good at.
Cons. Two databases to operate. A replication pipeline to maintain. Consistency window between the two. Doubled monitoring surface.

Does the team have the operational headcount for two systems?

5The 18-month horizon

The promise: don't revisit this choice for 18 months. At ~10M events/day growing 15%/quarter, the dataset hits ~7B rows by month 18.

Which option survives that growth without intervention?

6Hiring signal

The Postgres ecosystem has 10x more candidates than ClickHouse. If we ship with ClickHouse, our next two infra hires are constrained.

Does the hiring constraint change your read?

My recommendation: Option C (hybrid), with a caveat. Postgres handles config and state where consistency matters. ClickHouse handles event analytics where speed matters. The replication pipeline is a real cost — but cheaper than re-platforming at month 14 when Postgres aggregations start failing SLOs. Caveat: only if we hire one infra engineer with ClickHouse experience in the next 60 days. Otherwise default to Option A and accept the eventual migration.

+Your call

Which option, and what's the reasoning? If you're picking C, state the hiring commitment explicitly.
